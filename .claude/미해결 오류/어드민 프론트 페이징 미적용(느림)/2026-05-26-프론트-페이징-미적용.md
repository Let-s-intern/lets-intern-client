## 요약

아래 2개 화면은 **BE가 이미 페이징을 지원**(응답에 `pageInfo.totalElements` 포함)하는데,
**프론트가 그 페이징을 안 쓰고 전체를 받아서** 느리다.

- "크기 초과(413)"로 로드 실패하는 게 아니라 **레이턴시 문제**다 — 받아지긴 하는데 느리다.
- 따라서 **프론트만 수정**하면 즉시 개선된다. **BE 페이징 요청 대상이 아니다.**
- BE 페이징 자체가 없어 우회 불가한 케이스(후기 등)는 `[[어드민 후기 관리 페이징 미지원]]` 메모 참고. **이 문서와는 별개 이슈.**

| 화면 | 엔드포인트 | BE 페이징 | 프론트 현황 | 해결 주체 |
|---|---|---|---|---|
| 프로그램 관리 | `GET /api/v1/program/admin` | ✅ 지원 | `size=1000` 하드코딩 | **FE 단독** |
| 리드 히스토리 | `GET /api/v1/admin/lead-history` | ✅ 지원 | 전 페이지 while 루프 누적 | **FE 단독(단, 설계 고려 필요)** |

## 이슈 1: 프로그램 관리 (느림)

### 측정 (운영)

| 구분 | 응답 건수 | 크기 | 응답 시간 |
|---|---|---|---|
| 현재 (`size=1000` 전체) | 320건 | 250KB | **1.09초** |
| 페이징 (`size=20`) | 20건 | 12.7KB | **0.14초** |

→ **약 7.5배 빠름.**

### 근본 원인 (size 1000 하드코딩 + 클라이언트 슬라이싱)

`apps/admin/src/domain/admin/pages/program/Programs.tsx:62-65`

```tsx
const { data, isLoading, error } = useGetProgramAdminQuery({
  page: 1,
  size: 1000,   // ← 전체를 한 번에 받음
});
```

- 화면에는 `paginationModel`(`:67-68`)로 페이지네이션 UI가 있지만, DataGrid가 `paginationMode="server"` 없이(`:394-405`) **받은 전체를 클라이언트에서 잘라 보여주기만** 한다.
- 즉, 320건 전체를 매번 받아놓고 20건만 화면에 그린다 → 받는 1.09초가 통째로 낭비.

### BE 지원 근거 (이미 페이징됨)

- 응답에 `pageInfo.totalElements: 320` 존재. (`programAdminSchema` → `apps/admin/src/schema.ts:2112`, `pageInfo` → `:28-35`)
- 훅 `useGetProgramAdminQuery`(`apps/admin/src/api/program.ts:171-193`)는 **이미 `page`/`size` 파라미터를 받아 그대로 쿼리스트링에 전달**한다. 호출부만 바꾸면 끝.

### 수정 가이드 (FE 단독)

1. `Programs.tsx:62-65` — 하드코딩을 페이지네이션 모델 연동으로 교체:
   ```tsx
   const { data, isLoading, error } = useGetProgramAdminQuery({
     page: paginationModel.page + 1,        // 서버는 1-base
     size: paginationModel.pageSize,
   });
   ```
   (`paginationModel` 선언을 `useGetProgramAdminQuery` 호출보다 위로 올려야 함 — 현재 `:67`에 있음.)
2. DataGrid(`:394-405`)에 서버 페이징 옵션 추가:
   ```tsx
   paginationMode="server"
   rowCount={data?.pageInfo.totalElements ?? 0}
   ```
3. 필터(type/classification/status/날짜)도 함께 서버 파라미터로 넘기는지 확인 — 훅은 이미 해당 인자를 받으므로 클라이언트 필터링 흔적이 있으면 서버 위임으로 정리.

### 본보기 (이미 server 페이징 적용된 화면 — 복붙 대상)

- `apps/admin/src/domain/admin/pages/user/AdminUsersPage.tsx` — `paginationModel.page + 1`을 API `page`로 전달(`:10-15`).
- `apps/admin/src/domain/admin/user/users/table-content/UserAdminTable.tsx:424-432` — `paginationMode="server"` + `paginationModel` + `rowCount={totalElements}` 적용 완료.

## 이슈 2: 리드 히스토리 (매우 느림 — FE 전 페이지 루프)

### 측정 (운영)

- **전체 29,629건** (`pageInfo.totalElements: 29629`).
- 단일 페이지(`size=1000`) 1회: 445KB · 2.1초.
- 프론트가 **모든 페이지를 순차 루프**로 받음 → 약 30회 요청, 누적 ~13MB, **추정 60초+**.

### 근본 원인 (while 루프로 전 페이지 누적)

`apps/admin/src/api/lead.ts` — `useLeadHistoryListQuery`(`:74-115`)
※ 태스크 지시의 `useGetLeadHistoryListQuery`는 오기. 실제 이름은 **`useLeadHistoryListQuery`**.

```ts
const { pageSize = 1000, ...rest } = params;   // size 기본 1000

const leadHistoryList: LeadHistory[] = [];
let page = 1;
let totalPages = 1;

while (page <= totalPages) {                    // ← 전 페이지 순회
  const res = await axios.get('/admin/lead-history', {
    params: { page, size: pageSize, /* ...필터 */ },
  });
  const parsed = leadHistoryListSchema.parse(res.data.data);
  leadHistoryList.push(...parsed.leadHistoryList);   // ← 누적
  totalPages = Math.max(parsed.pageInfo.totalPages, 1);
  if (!parsed.leadHistoryList.length) break;
  page += 1;
}
return leadHistoryList;
```

- 호출부 `apps/admin/src/domain/admin/lead/hooks/useLeadHistoryData.ts:13`은 인자 없이 호출 → 매번 29,629건 전체 fetch.

### BE 지원 근거

- 엔드포인트가 `page`/`size`/`totalElements` 페이징을 정상 지원함(`leadHistoryListSchema`에 `pageInfo` 포함, `lead.ts:34-37`).
- 즉 BE는 멀쩡하고, **프론트 루프가 페이징을 일부러 무력화**하는 구조.

### ⚠️ 주의 — 단순 서버 페이징으로 바꾸면 기능이 깨진다 (설계상 전체 데이터 의존)

태스크 초안의 "while 루프 제거 → 1페이지씩만 요청 + DataGrid `paginationMode=server`"는 **그대로 적용하면 안 된다.** 이 화면은 MUI DataGrid가 아니라 전체 데이터 클라이언트 가공에 의존한다:

- 목록 테이블은 MUI DataGrid가 아니라 `@tanstack/react-table`을 쓰며, 이미 **클라이언트에서 상위 100건만 렌더**(`apps/admin/src/domain/admin/lead/ui/LeadHistoryTable.tsx:11,32-34` — `VISIBLE_ROW_LIMIT = 100`).
- `useLeadHistoryData.ts`가 받은 전체를 **전화번호별 1행으로 집계**(`aggregatedRows`, `:124-170`)하고, **마그넷/프로그램/마그넷타입 옵션과 그룹 요약을 전체 스캔으로 생성**(`:59-121`).
- `useLeadHistoryFilter`는 그 전체 집합에 **AND/OR 트리 필터**를 적용.
- CSV 내보내기(`LeadHistoryPage.tsx:54-56` → `utils/csv.ts:downloadLeadHistoryCsv`)는 **필터링된 집계 전체**를 내려받는다.

즉 "목록 1페이지만"으로 바꾸면 집계·필터·CSV가 부분 데이터만 보고 잘못 동작한다. **전체 루프는 (의도가 명시돼 있진 않지만) 이 집계/필터/export 설계를 떠받치는 사실상의 전제**다.

### 수정 가이드 (FE 단독, 경로 분리 방향)

목록 표시용과 전체 집계/export용을 **분리**하는 방향으로 간다. 둘 중 하나 선택:

**A안 — 목록은 서버 페이징, 집계/export는 별도 경로 (권장 장기 방향)**
- 목록 표시: `useLeadHistoryListQuery`에서 while 루프 제거 → 서버 페이징 1페이지씩. 테이블을 `paginationMode="server"` + `rowCount={totalElements}`로(현재 react-table이면 MUI DataGrid로 교체 또는 react-table 서버 페이징 모델 적용).
- 전체 집계(전화번호 그룹·옵션·필터)와 CSV export: 전체가 꼭 필요한 부분만 별도 "전체 로드" 경로로 분리하고, 버튼 클릭 시점에만(lazy) 전체를 받게 한다. → 페이지 진입 즉시 60초 멈춤 제거.
- 집계/필터를 서버로 위임할 수 있으면(전화번호 그룹 집계 API 등) FE 부담이 가장 줄지만, 그건 BE 협의 필요 — **이 문서 범위(FE 단독) 밖**이라 옵션으로만 둔다.

**B안 — 최소 변경(즉시): 전체 fetch는 유지하되 진입 시 자동 실행 안 함**
- 페이지 진입 시 자동으로 29,629건을 받지 않게 하고(예: "리드 불러오기" 버튼 / lazy `enabled`), CSV·필터가 실제 필요할 때만 전체 로드.
- 루프 구조는 유지하므로 집계/필터/CSV 기능은 그대로 동작. 페이지 첫 진입 60초 멈춤만 제거하는 응급 처치.

> 어느 쪽이든 **"목록 보기 = 한 페이지", "집계/필터/CSV = 전체 로드"** 두 경로를 나누는 것이 핵심.
> 관련 파일: `LeadHistoryPage.tsx`, `ui/LeadHistoryTable.tsx`, `hooks/useLeadHistoryData.ts`, `hooks/useLeadHistoryFilter.ts`, `utils/csv.ts`, `api/lead.ts`.

## 공통 결론

- 두 건 모두 **413(크기 초과)이 아니라 레이턴시 문제**이며, **프론트 단독 수정으로 즉시 개선 가능**하다. BE 페이징은 이미 있다.
- **이슈 1(프로그램)**: 호출부 한 줄 + DataGrid 옵션 2개. user 화면 본보기 복붙. 리스크 낮음, 7.5배 개선. **우선 처리 권장.**
- **이슈 2(리드)**: 단순 서버 페이징 전환은 집계/필터/CSV를 깨뜨림. **목록용 / 전체-집계용 경로 분리**가 핵심. 응급으로는 진입 시 자동 전체 fetch 제거(B안)부터.
- BE 페이징이 아예 없어 프론트로 우회 불가한 케이스(후기 관리 등)는 `[[어드민 후기 관리 페이징 미지원]]`에서 다룸 — **본 문서와 별개**.
