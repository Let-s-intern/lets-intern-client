# 어드민 후기 관리 — 페이징 미지원으로 대용량 응답 로드 실패

> **상태**: 🔴 미해결 (백엔드 수정 필요 — 프론트 단독 해결 불가)
> **작성일**: 2026-05-26
> **영향 화면**: 어드민 > 후기 관리 (챌린지/라이브/미션/리포트/블로그 후기 전체)
> **분석 대상**: `workspace/lets-career-server` (Java Spring + QueryDSL), `workspace/lets-intern-client` (apps/admin)

---

## 요약

어드민 후기 관리 목록 API가 **페이징을 지원하지 않고 전체 후기를 한 번에 반환**한다.
후기 데이터가 쌓이면서 응답 크기가 임계점을 넘어 **로드 자체가 실패**하는 상태.

- 화면(MUI DataGrid)에 페이지네이션 UI는 있지만, 실제 서버 요청에 `page/size`가 전달되지 않음 → **클라이언트가 받은 전체를 메모리에서 자르는 흉내**만 가능.
- 응답이 통째로 너무 커서 그 "받기" 단계에서 실패하므로, **프론트의 어떤 트릭으로도 우회 불가** (받을 데이터 자체가 손에 안 들어옴).
- 근본 해결 = 백엔드가 `Pageable`을 받아 일부만 잘라 반환하도록 수정.

## 이미 있는 페이징 본보기 (복붙 대상)

새 패턴 도입이 아니라 **같은 코드베이스의 검증된 패턴을 복사**하면 된다.

- `PageInfo` 유틸: `global/common/entity/PageInfo.java` — `PageInfo.of(Page<?>)`.
- **블로그 후기(유저용) — 거의 동일 도메인**: `BlogReviewServiceImpl.java:53-57`
  ```java
  public GetBlogReviewResponseDto getBlogReviews(List<ProgramType> typeList, Pageable pageable) {
      Page<BlogReviewVo> blogReviewVos = blogReviewHelper.findAllBlogReviewVos(typeList, pageable);
      PageInfo pageInfo = PageInfo.of(blogReviewVos);
      return reviewMapper.toGetBlogReviewResponseDto(blogReviewVos.getContent(), pageInfo);
  }
  ```
  대응 리포지토리 `BlogReviewQueryRepositoryImpl.java:52` `findAllBlogReviewVos(typeList, pageable)` — `.offset().limit()` + count 쿼리로 `Page<>` 반환. **블로그 후기 어드민에 붙일 페이징 쿼리가 같은 파일 안에 이미 존재.**
- 어드민에서 이미 페이징하는 다른 도메인: lead / notice 목록 / user 목록 / usercareer / userexperience.

## BE 수정 가이드

### A. 프로그램 후기 `/admin/review/{type}` (챌린지/라이브/미션/리포트 한 번에 해결)

1. **컨트롤러** `ReviewV2AdminController.getReviewsForAdmin` — 시그니처에 `final Pageable pageable` 추가, 서비스로 전달.
2. **서비스 IF** `ReviewService.getReviewsForAdmin()` → `getReviewsForAdmin(Pageable pageable)`.
3. **서비스 구현 4종** — `List` 처리 → `Page<>` 처리 후 `PageInfo.of(page)` 생성.
4. **리포지토리 IF/구현 3~4종** (`ChallengeReviewQueryRepository(Impl)`, `Live...`, `Report...`, 미션 해당분) — `List<...AdminVos>` → `Page<...AdminVos>`. `.offset(pageable.getOffset()).limit(pageable.getPageSize())` + count 쿼리 (`PageableExecutionUtils.getPage`).
5. **응답 DTO** `GetReviewForAdminResponseDto` — `PageInfo pageInfo` 필드 추가 (총 페이지 표시 안 할 거면 생략 가능).
6. **Mapper** `ReviewMapper.toGetReviewForAdminResponseDto` — `pageInfo` 인자 추가.

### B. 블로그 후기 `/admin/review/blog`

1. **컨트롤러** `getBlogReviewsForAdmin` — `Pageable` 추가.
2. **서비스** `BlogReviewServiceImpl.getBlogReviewForAdmin(Pageable)` — `findAllBlogReviewAdminVos()`를 페이징 버전으로 교체(같은 파일 `findAllBlogReviewVos` 패턴 복사).
3. **리포지토리** `BlogReviewQueryRepositoryImpl.findAllBlogReviewAdminVos` — `List` → `Page` (`.offset().limit()` + count). 패턴은 같은 파일 `:52`에 존재.
4. **응답 DTO** `GetBlogReviewForAdminResponseDto` — `pageInfo` 추가.

### 같이 수정하기 쉬운 묶음 (권장 PR 단위)

- **A를 먼저** — 한 엔드포인트 수정으로 **챌린지+라이브+미션+리포트 4개 화면이 동시 해결**되어 가성비 최고.
- **B는 가장 기계적으로 쉬움** — 붙일 페이징 쿼리가 같은 파일에 이미 있어 거의 복붙.
- A·B 모두 `ReviewV2AdminController` 한 파일에서 만나고 `PageInfo`를 공유 → **한 PR로 묶는 게 자연스러움.**

## 결론

- 챌린지·블로그 후기 **둘 다 페이징 미적용**. 어드민 후기 관리 5개 화면 전부 해당.
- 프론트로는 우회 불가(응답 자체가 커서 로드 실패). **BE에 `Pageable` 추가가 유일한 근본 해결.**
- 본보기 코드(유저 블로그 후기·lead·user)가 충분해 작업 난이도·리스크는 낮음.
- 권장: 프로그램 후기(A) + 블로그 후기(B)를 한 PR로, A부터.
