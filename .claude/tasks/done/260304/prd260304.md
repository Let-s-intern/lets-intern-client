## 1. 개요

멘토 전용 마이페이지를 구축한다. v0.1.1 범위는 **멘토 화면 3개** (프로그램 일정 / 서면 피드백 모달 / 프로필)이다.

### 1-1. 핵심 원칙

- 기존 공통 컴포넌트(`BaseModal`, `Button`, `Input` 등)를 최대한 활용
- **현재 구현된 API만 우선 연결**, 신규 API가 필요한 부분은 UI/UX만 선 구현 (클릭 불가 OK)
- 기존 컨벤션 준수: React Query + Zod 스키마, Tailwind CSS, App Router
- 도메인 폴더 구조: `domain/mentor/` 신규 생성

### 1-2. 아키텍처 원칙: 멘토 도메인 독립 분리

멘토는 user, admin과 동급의 **핵심 유저 타입**으로, 앞으로 기능이 지속 확장될 예정이다.
따라서 `(user)` 라우트 그룹에 종속시키지 않고, admin과 동일한 패턴으로 완전히 독립된 도메인으로 분리한다.

```
app/
├── (user)/     # 일반 유저 (취준생)
├── admin/      # 관리자
└── mentor/     # 멘토 ← user 하위가 아닌 독립 라우트
```

**분리 근거**:
- 멘토는 고유한 레이아웃 (사이드바 + 캘린더)을 가짐 — user 레이아웃(ConditionalLayout + Header/Footer)과 다름
- 멘토 전용 Provider, Guard가 필요 — admin 패턴과 동일
- 향후 멘토 대시보드, 정산, 알림 등 기능 확장 시 독립 구조가 필수
- API 레이어도 `api/mentor/`로 분리하여 멘토 전용 훅/스키마를 독립 관리

**참고 — admin 구조 (동일 패턴 적용)**:
```
app/admin/layout.tsx → AdminProviders > AdminGuard > AdminSidebar + children
context/AdminProviders.tsx → 독립 QueryClient + LocalizationProvider
```

## 4. 화면별 상세 스펙

### 4-1. S01 — 프로그램 일정 (메인 화면)

**경로**: `/mentor`

#### 레이아웃
- **좌측 사이드바**: Logo + 네비게이션 (프로그램 일정, Profile)
- **우측 메인**: 환영 메시지 → 주간 요약 → 주 내비게이션 + 필터 → 주간 캘린더

#### WK-07 환영 메시지
- 상단에 텍스트 표시: `"{멘토명} 멘토님, 렛츠커리어와 함께 취준생들의 고민에 귀 기울여 주셔서 언제나 감사합니다. 앞으로도 잘 부탁드립니다!"`
- v0.1.1에서는 하드코딩 (신규 API 미구현)
- 멘토명은 `GET /api/v1/user` 또는 `GET /api/v2/user/me` 응답의 name 필드로 치환

#### WK-03 주간 요약 카드
- 카드 3개: `이번주 전체` / `오늘 마감` / `미완료`
- 기존 API 데이터 집계로 계산:
  - `이번주 전체`: 해당 주에 기간이 겹치는 전체 피드백 건수 (attendanceList 전체 count)
  - `오늘 마감`: feedbackEndDate가 오늘인 미션의 attendanceList count
  - `미완료`: feedbackStatus가 COMPLETED/CONFIRMED이 아닌 건수

#### WK-02 주 내비게이션
- `← MM.DD ~ MM.DD →` 형태
- 좌/우 화살표로 이전/다음 주 이동
- 기본값: 현재 주 (월요일 시작)

#### WK-04 챌린지 필터
- `GET /api/v1/challenge-mentor` 응답의 챌린지 목록으로 칩 렌더링
- 칩 클릭 → 해당 챌린지만 캘린더에 표시
- "전체" 기본 선택

#### WK-01 주간 캘린더
- 월~일 7일 칼럼
- 오늘 날짜: 파란 원 하이라이트
- 각 챌린지의 미션별 피드백 기간을 바(bar)로 표시

#### WK-05 챌린지 기간 바
- 바 표시 정보: `[서면]` 라벨 + 챌린지명 + 회차 피드백 기간
- 기간: feedbackStartDate ~ feedbackEndDate (미션 응답에서 추출)
- 하단 텍스트: `미제출 N / 제출 N` + `시작전 N / 진행 중 N / 완료 N`
- **서면 바 색상**: 베이지 (#FFF3E0 계열)

#### WK-06 바 클릭 → 모달
- 클릭 시 `FeedbackModal` 오픈
- challengeId, missionId를 props로 전달

### 4-2. S02 — 서면 피드백 모달

**트리거**: 프로그램 일정에서 기간 바 클릭

#### 레이아웃 (BaseModal 확장)
- 크기: `w-[1060px] h-[700px]` (와이어프레임 기준, 반응형 조정)
- 좌측 (약 30%): 챌린지/회차 헤더 + 멘티 리스트
- 우측 (약 70%): 상태 인디케이터 → 멘티 정보 → 에디터 → 액션 버튼

#### FB-06 상태 인디케이터
- `시작 전 → 진행 중 → 완료` 3단계 스텝 바
- 현재 상태에 색상 적용: 시작 전(회색), 진행 중(노랑/주황), 완료(초록)
- feedbackStatus 값으로 결정: WAITING → IN_PROGRESS → COMPLETED

#### FB-01 멘티 리스트
- 그룹 헤더: `{챌린지명} {N}차 피드백`
- 멘티 항목: `이름 + 상태 뱃지`
- 뱃지 색상:
  - 제출전 (link 없음): 빨강
  - 시작전 (WAITING): 회색
  - 진행 중 (IN_PROGRESS): 노랑
  - 완료 (COMPLETED): 초록

#### FB-02 멘티 정보
- 멘티명, 챌린지명 + 회차
- 희망 직군: `wishJob` 필드
- 희망 산업: 스키마에 필드 없으면 빈칸 (UI만 표시)
- 희망 기업: `wishCompany` 필드

#### FB-03 Markdown 에디터
- 기존 Lexical 에디터 활용 (`domain/admin/lexical/`)
- FB-02 응답의 `feedback` 필드로 초기값 로드 (null이면 빈 에디터)
- Markdown 입출력 지원

#### FB-04 임시 저장
- `PATCH /api/v1/attendance/{aId}/mentor`
- Body: `{ feedback: "에디터 내용", feedbackStatus: "IN_PROGRESS" }`
- 성공 시: 좌측 뱃지 노랑으로 갱신 + 토스트 알림

#### FB-05 최종 제출
- 클릭 시 확인 모달 표시: "최종 제출하시겠습니까? 제출 후에는 수정할 수 없습니다."
- 확인 → `PATCH /api/v1/attendance/{aId}/mentor`
- Body: `{ feedback: "에디터 내용", feedbackStatus: "COMPLETED" }`
- 성공 시: 좌측 뱃지 초록으로 갱신

#### FB-07 닫기 / 멘티 전환
- X 버튼: 미저장 변경 있으면 확인 모달 → 닫기
- 멘티 클릭 전환: 미저장 변경 있으면 확인 모달 → 전환 시 FB-02 재호출
- 미저장 감지: 에디터 현재 내용 vs 서버에서 받은 feedback 비교

- 기존 `AdminGuard`는 isAdmin || isMentor로 둘 다 허용 → MentorGuard는 isMentor만 체크
- layout.tsx에서 `MentorProviders > MentorGuard > Sidebar + children` 구조

### 6-2. MentorProviders (AdminProviders 동일 패턴)

- `context/MentorProviders.tsx`
- 독립 QueryClient (admin과 캐시 분리)
- 멘토 전용 Context가 필요하면 여기에 추가 (e.g. MentorSnackbarProvider)

### 6-3. 기존 AdminGuard 영향

- 현재 AdminGuard는 `isAdmin || isMentor`로 멘토도 admin 접근 허용 중
- mentor 독립 분리 후에도 이 로직은 유지 (멘토가 어드민 피드백 기능 사용 가능)
- 향후 권한을 분리할 때 AdminGuard에서 isMentor 제거 가능

## 8. 주요 유저 플로우

### 8-1. 멘토 피드백 수행

```
로그인 → /mentor 진입 → is-mentor 확인
→ 주간 요약 확인 (이번주 전체/오늘마감/미완료)
→ 챌린지 필터 (선택)
→ 캘린더에서 기간 바 클릭
→ 서면 피드백 모달 팝업
→ 좌측에서 멘티 선택
→ 노션 링크로 미션 제출물 확인 (새 탭)
→ Markdown 에디터로 피드백 작성
→ 임시 저장 or 최종 제출
→ 다른 멘티 선택 → 반복
→ X로 모달 닫기
```

### 8-2. 모달 인터랙션 상세

```
[모달 열림]
→ 첫 번째 멘티 자동 선택 → FB-02 호출

[멘티 전환]
→ 에디터 변경 있음? → 확인 모달 "저장하지 않고 전환하시겠습니까?"
  → 확인: FB-02 재호출 (새 멘티)
  → 취소: 현재 멘티 유지

[임시 저장]
→ PATCH 호출 → 성공 → 뱃지 노랑 갱신 → 토스트 "임시 저장되었습니다"

[최종 제출]
→ 확인 모달 → PATCH 호출 → 성공 → 뱃지 초록 갱신 → 에디터 readOnly

[닫기]
→ 에디터 변경 있음? → 확인 모달 → 닫기
```

## 9. 주간 캘린더 렌더링 로직

### 9-1. 데이터 조합 과정

```
1. GET /api/v1/challenge-mentor → 참여 챌린지 목록 [{challengeId, title, ...}]
2. 각 챌린지별 GET /api/v1/challenge/{cId}/mission/feedback → 미션 목록 [{missionId, th, startDate, endDate, ...}]
3. 각 미션별 GET /api/v1/challenge/{cId}/mission/{mId}/feedback/attendances → 참여자 목록
4. 참여자 feedbackStatus로 카운트 계산
```

### 9-2. 기간 바 배치 알고리즘

```
- 미션의 feedbackStartDate ~ feedbackEndDate 범위가 현재 주(weekStart ~ weekEnd)와 겹치면 표시
- 바는 시작일 칼럼부터 마감일 칼럼까지 span
- 주 경계를 넘어가면 해당 주 범위만큼 잘라서 표시
- 여러 바가 같은 날짜에 겹치면 수직 스택
```

### 9-3. 참고: feedbackStartDate/feedbackEndDate

- 현재 미션 API 응답에 `startDate`, `endDate` 필드 존재
- 이것이 피드백 기간인지 미션 기간인지 확인 필요
- 없으면 미션 startDate~endDate를 임시로 사용

## 10. 기술적 고려사항

### 10-1. Lexical 에디터 재사용

- 기존 `domain/admin/lexical/` 에디터를 멘토 피드백에서도 사용
- 필요한 플러그인만 선별 (Markdown shortcut, 기본 서식)
- 에디터 초기값: `$convertFromMarkdownString(feedback)`
- 에디터 출력: `$convertToMarkdownString()`

### 10-2. API 호출 최적화

- 챌린지 목록 → 미션 목록 → 참여자 목록 순차 호출이 많아 waterfall 발생 가능
- `Promise.all`로 챌린지별 미션 목록 병렬 호출
- attendances는 모달 오픈 시점에만 호출 (캘린더에서는 카운트만 필요)
- React Query의 `staleTime` 활용: 같은 주 데이터는 캐싱

### 10-3. 날짜 유틸리티

- `date-fns` 사용 (프로젝트에 이미 설치됨)
- `startOfWeek(date, { weekStartsOn: 1 })` — 월요일 시작
- `format`, `addWeeks`, `subWeeks`, `isWithinInterval`, `isSameDay`
