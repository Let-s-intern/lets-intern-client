# Troubleshooting — 원인 미상·미해결 이슈 추적 로그

원인을 아직 못 찾았거나, 알면서도 지금 고치지 못한 이슈를 한곳에 모아 **단서·가설·시도**를
잃지 않고 추적한다. "언젠가 다시 마주칠 문제"를 매번 처음부터 파지 않기 위한 곳이다.

**경계 (여기 두지 않는 것)**
- 해결된 버그의 수정 내용 → 커밋 메시지/PR에 남는다.
- 지금 진행 중인 작업 → `../tasks/` (todo/done).
- 여기엔 **원인 미상이거나 보류된** 문제만 둔다.

## 언제 추가하나

- 재현은 되는데 원인을 못 찾았을 때
- 원인은 짐작되지만 지금 고치지 않고 보류할 때
- 산발적으로만 재현되는(flaky) 문제
- 외부(서드파티·BE·인프라) 원인 의심으로 막힌 문제

## 파일 규칙

- 한 이슈 = 한 파일. 파일명은 `<영역>-<짧은-증상>.md` (예: `chat-소켓-끊김.md`, `pdf-한글-깨짐.md`).
- 새 파일을 만들면 아래 **인덱스** 표에 한 줄 추가한다.
- 해결되면 파일의 "결론" 섹션을 채우고 상태를 🟢 로 바꾼다 (파일은 지우지 말고 남겨 — 같은 문제 재발 시 근거).

## 이슈 파일 템플릿

아래를 복사해 새 파일로 저장한다.

````markdown
# <증상 한 줄 요약>

- 상태: 🔴 미해결 | 🟡 보류 | 🟢 해결(결론 참조)
- 영역: <chat / pdf / build / auth / …>
- 처음 관측: YYYY-MM-DD
- 재현 빈도: 항상 / 가끔 / 1회

## 증상
무엇이 어떻게 잘못되는가 (정확한 에러 메시지·로그·스크린샷 경로 포함).

## 재현 절차
1.
2.

## 환경
브랜치/커밋, OS, 브라우저, Node/패키지 버전 등 — 재현되는 조건과 안 되는 조건.

## 가설 → 검증 로그
- [ ] 가설1: …  → 검증: …  → 결과: 반증 / 지지 / 미확인
- [ ] 가설2: …

## 현재 상태 / 다음 시도
막힌 지점, 다음에 해볼 것, 도움 필요한 부분.

## (해결 시) 결론
- 근본 원인:
- 해결/우회:
- 관련 커밋·PR:
````
