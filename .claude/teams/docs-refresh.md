# Docs Refresh Team

대규모 코드 변경(모노레포 전환·도메인 분리·주요 리팩터) 직후 **문서 stale** 을 해소하는 팀. 단일 `doc-updater` 에이전트로도 충분하지만, 영역이 넓으면 `doc-updater` + `doc-worker`로 분할.

## 팀원 구성

| 역할 | 에이전트 타입 | 인원 | 담당 |
|---|---|---|---|
| 팀 리드 | 메인 세션 | 1 | 변경 범위 파악·결과 검수 |
| 문서 업데이트 | `doc-updater` | 1 | 기존 문서의 경로·네이밍·섹션 최신화 |
| 신규 문서 작성 | `doc-worker` | 0~1 | 배포 가이드·BE 요청·아키텍처 등 신규 필요 시 |

## 병렬 가능성

`doc-updater`와 `doc-worker`는 서로 다른 파일을 건드리므로 병렬 실행 가능. 단 CLAUDE.md 같은 인덱스 파일은 리드가 직접 수정.

## Spawn Prompt 템플릿

```text
`.claude/docs/` 하위 문서를 <기준 날짜> 기준으로 최신화해주세요.

**최신화 맥락**:
- <주요 변경 1>: <bullet>
- <주요 변경 2>: <bullet>

**최신화 대상**:
- <파일 경로>: <무엇을 확인·반영>
- ...

**규칙**:
1. 실제로 바뀐 내용만 반영 — 기존에 정확한 서술은 유지
2. 경로 참조가 구 구조라면 신규 경로로 교체 (apps/*, packages/*)
3. 신규 파일 생성 금지 — 기존 파일 업데이트만
4. emoji 사용 금지
5. 한국어로 작성
6. 변경 없을 파일은 건드리지 않음

**참조**:
- CLAUDE.md
- 관련 PRD 파일
- 실제 apps/·packages/ 구조

**주의**:
- 현재 apps/<이식 중인 앱>/ 소스는 백그라운드 작업 중이라 상태 불안정 → 다른 참조 경로 사용
- 변경된 파일 리스트와 요약만 보고 (200자 이내)
```

## 완료 기준

- 대상 문서 경로가 모두 신규 구조 반영
- 날짜·버전 정보 업데이트
- 리드가 diff 검토 후 커밋

## 주의

- 문서는 코드와 달리 **stale해도 빌드가 실패하지 않음** → 리드가 적극 검수
- 같은 내용이 여러 곳에 중복되지 않도록 **단일 출처(Single Source of Truth)** 원칙 유지
- CLAUDE.md는 짧은 index 역할만, 상세 규칙은 `.claude/skills/*/SKILL.md`로 위임
