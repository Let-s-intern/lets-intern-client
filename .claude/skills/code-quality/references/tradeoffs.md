# 기준 간 트레이드오프

4가지 기준을 모두 한꺼번에 충족하기는 어렵다.
현재 상황을 바탕으로 어떤 가치를 우선해야 하는지 판단해야 한다.

---

## 응집도 ↔ 가독성

공통화/추상화하면 응집도는 높아지지만, 코드가 한 차례 추상화되어 가독성이 떨어진다.

| 상황                                | 우선하는 기준 | 행동           |
| ----------------------------------- | ------------- | -------------- |
| 함께 수정하지 않으면 오류 발생 가능 | **응집도**    | 공통화, 추상화 |
| 위험성이 높지 않은 경우             | **가독성**    | 코드 중복 허용 |

```tsx
// 예: 챌린지 가격과 라이브 가격의 할인 계산 로직이 같음

// 이 로직이 틀리면 결제 금액이 잘못됨 → 응집도 우선
function calculateDiscount(price: number, coupon: Coupon): number { ... }
// → 한 곳에서 관리

// 챌린지 카드와 블로그 카드의 날짜 표시가 비슷함
// 하지만 향후 다르게 변경될 수 있음 → 가독성 우선
function ChallengeCard() { return <span>{formatDate(date)}</span>; }
function BlogCard() { return <span>{formatDate(date)}</span>; }
// → 중복 허용
```

## 결합도 ↔ 응집도

중복 코드를 허용하면 영향범위가 줄어서 결합도가 낮아진다. 하지만 한쪽을 수정했을 때 다른 한쪽을 실수로 수정하지 못할 수 있어 응집도가 떨어진다.

| 상황                                      | 우선하는 기준 | 행동                  |
| ----------------------------------------- | ------------- | --------------------- |
| 비즈니스 규칙이 반드시 동일해야 함        | **응집도**    | 공통 함수/상수로 추출 |
| UI가 비슷하지만 독립적으로 변경될 수 있음 | **결합도**    | 중복 코드 허용        |

## 실전 판단 플로우

```
코드가 비슷한 곳이 여러 군데 있다
  ├─ 비즈니스 규칙이 반드시 동일해야 하는가?
  │   ├─ YES → 공통화 (응집도 우선)
  │   └─ NO → 독립적으로 변경될 수 있는가?
  │       ├─ YES → 중복 허용 (결합도 우선)
  │       └─ NO → 핵심 도메인 폴더에 두기
  │
  └─ 추상화 수준을 높여야 하는가?
      ├─ 추상화하지 않으면 버그 위험이 큰가?
      │   ├─ YES → 추상화 (응집도 우선)
      │   └─ NO → 구체적으로 유지 (가독성 우선)
      └─ 지금 3곳 이상에서 쓰이는가?
          ├─ YES → common/hooks/utils로 이동
          └─ NO → 도메인 폴더에 유지
```

## 핵심 원칙

> 장기적으로 코드가 수정하기 쉽게 하기 위해 어떤 가치를 우선해야 하는지 고민해야 한다.
> 정답은 없다. 상황에 따라 다르다. 중요한 것은 **의식적으로 판단**하는 것이다.
