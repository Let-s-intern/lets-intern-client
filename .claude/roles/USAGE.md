# 에이전트 팀 사용 가이드

이 프로젝트는 **에이전트 팀** 기능을 사용하도록 설정되어 있습니다.

## 에이전트 팀이란?

- **여러 개의 독립적인 Claude Code 세션**이 하나의 팀으로 협업
- 각 팀원은 **자신만의 터미널/패널**을 가짐
- 팀원끼리 **직접 메시지를 주고받으며** 협업
- **공유 태스크 리스트**로 작업 조율
- 사용자가 **실시간으로 개입**하여 방향 수정 가능

## 현재 설정

### 1. 환경 설정 (이미 완료됨)

`.claude/settings.json`:
```json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  },
  "teammateMode": "tmux"
}
```

### 2. 팀 구성

- **coordinator** - 팀 리드 (사용자 명령 수신 및 작업 할당)
- **developer** - 개발 팀원 (자율적 코드 구현)
- **tester** - 테스트 팀원 (ESLint, Prettier, 유닛 테스트, UI 수정)

## 사용 방법

### 1단계: Coordinator로 Claude Code 시작

```bash
claude
```

일반적으로 Claude Code를 시작하면 됩니다.

### 2단계: 에이전트 팀 생성 요청

Claude에게 자연어로 에이전트 팀 생성을 요청합니다:

```
에이전트 팀을 만들어줘. 3명으로 구성할게:

1. 나는 coordinator 역할로 명령을 내릴거야
2. developer 팀원 1명 (코드 구현 담당)
3. tester 팀원 1명 (테스트 및 UI 수정 담당)

.claude/roles/ 디렉토리에 각 역할의 설정이 있어. 그걸 참고해서 팀원을 생성해줘.
```

### 3단계: 팀원에게 작업 할당

Coordinator로서 팀원들에게 작업을 할당합니다:

```
Developer에게: "로그인 페이지 구현해줘. .claude/docs/letscareer/ 컨벤션 따라서"

Tester에게: "로그인 페이지 완성되면 ESLint, Prettier, 타입 체크 돌려줘"
```

### 4단계: 실시간 방향 조정

팀원이 작업 중일 때 언제든 개입 가능:

```
Developer야, 잠깐! OAuth 대신 이메일 로그인으로 바꿔줘
```

### 5단계: 팀원과 직접 대화

- **In-process 모드**: `Shift+Down`으로 팀원 전환 후 메시지 입력
- **Tmux 모드**: 팀원 패널 클릭하여 직접 대화

### 6단계: 팀 정리

작업 완료 후:

```
팀을 정리해줘 (Clean up the team)
```

## 디스플레이 모드

### Tmux 모드 (현재 설정)
- 각 팀원이 **별도 패널**에 표시
- **동시에 모든 팀원의 작업** 확인 가능
- 패널 클릭으로 팀원과 직접 대화

### In-process 모드
- 모든 팀원이 **하나의 터미널** 안에서 실행
- `Shift+Down`으로 팀원 전환
- tmux 없이 사용 가능

설정 변경:
```json
{
  "teammateMode": "in-process"  // 또는 "tmux"
}
```

## 주요 차이점: 에이전트 팀 vs 서브에이전트

| 특징 | 에이전트 팀 (현재 설정) | 서브에이전트 |
|------|-------------------------|--------------|
| 세션 | 각자 독립적인 세션 | 단일 세션 내부 |
| 커뮤니케이션 | 팀원끼리 직접 메시지 | 메인에게만 보고 |
| 작업 조율 | 공유 태스크 리스트 | 메인이 관리 |
| 사용자 개입 | 언제든 팀원에게 직접 | 메인을 통해서만 |
| 토큰 비용 | 높음 (독립 세션) | 낮음 (단일 세션) |

## 워크플로우 예시

### 새 기능 개발

```
사용자: "사용자 프로필 편집 기능 추가해줘"

→ Coordinator가 작업 분석
→ Developer에게: "프로필 편집 폼 컴포넌트 구현"
→ Developer 작업 완료 보고
→ Tester에게: "프로필 편집 기능 테스트"
→ Tester 테스트 완료 보고
→ Coordinator가 사용자에게 최종 보고
```

### 방향 수정 시나리오

```
사용자: "다크모드 추가해줘"
→ Developer가 CSS 변수 방식으로 작업 중...

사용자: "아니야, styled-components ThemeProvider 사용해"
→ Coordinator가 Developer에게 방향 수정 전달
→ Developer가 새로운 방식으로 재구현
→ Tester가 검증
```

## 팀원 설정 파일

각 역할의 상세 설정은 다음 파일 참조:
- `coordinator.md` - 팀 리드 역할 및 작업 할당 전략
- `developer.md` - 개발 워크플로우 및 코드 품질 기준
- `tester.md` - 테스트 프로세스 및 UI 수정 가이드

## 제약사항

- **한 번에 하나의 팀만** 관리 가능
- **세션 재개 시** in-process 팀원은 복구되지 않음
- **팀원은 팀을 생성할 수 없음** (리드만 가능)
- **VS Code 통합 터미널**에서는 split-pane 모드 미지원
